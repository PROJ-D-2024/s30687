# Assignment 4 technical description

## 1. Goal of the technical setup

The purpose of this repository is not only to store code, but to define a controlled implementation environment for the thesis experiment described in Assignment 3. The research question asks whether demographic structure improves the explanation of variation in hospital service use across GP practices in Scotland when compared with a simpler baseline based on practice size and GP availability. Assignment 4 therefore focuses on the engineering side of the thesis: the repository must be understandable to a third party, executable from documented commands, and structured in a way that makes future experiments traceable.

The design target was practical reproducibility. A supervisor or reviewer should be able to identify where raw inputs are placed, how they are transformed, which configuration produced a given result, and where the resulting metrics and model artifacts were written. This means that the repository must expose a clear pipeline, deterministic execution rules, and structured output storage.

## 2. Repository architecture and separation of concerns

The repository now follows a conventional data-science engineering layout:

- `data/raw/` contains immutable source files,
- `data/processed/` contains derived analytical tables generated by scripts,
- `src/` contains reusable Python modules and thin command wrappers,
- `config/` contains the global configuration and per-experiment overrides,
- `reports/` contains run-specific metrics and metadata,
- `models/` contains serialized model artifacts,
- `docs/` contains the reproducibility protocol and this technical description.

This separation is important because notebook-only or script-only workflows often hide intermediate steps. In the current structure, preprocessing logic is isolated in `src/data_pipeline.py`, model construction and evaluation live in `src/modeling.py`, experiment artifact management is centralized in `src/tracking.py`, and the command modules (`preprocess.py`, `train.py`, `evaluate.py`) serve as clean entry points rather than places where all logic is mixed together.

The repository also exposes a single execution pattern through `python -m src <command> --config ...`. This unifies how the project is run and removes ambiguity about which script should be called.

## 3. Data flow and preprocessing decisions

The current reproducibility scaffold uses the raw workbook `gp_practice_population_demographics_merged.xlsx` stored locally in `data/raw/`. During preprocessing, the pipeline reads the `data` worksheet, validates the required columns, parses dates, and converts the long-format demographic table into a one-row-per-practice-and-date analytical table. The `Sex == "All"` rows provide total population and age-band counts, while the `Sex == "Female"` rows are merged back in to derive a female population share.

From these inputs the preprocessing step calculates the core demographic variables required by the experiment design, including:

- `share_age_65_plus`,
- `share_age_0_14`,
- `share_female`.

At this stage the distributable repository does not contain the real hospital use outcome table, GP workforce data, or deprivation controls. Because the assignment still requires an end-to-end reproducible workflow, the preprocessing pipeline generates deterministic placeholder columns for:

- `gp_availability`,
- `deprivation_index`,
- `hospital_use_per_1000`.

These placeholders are not presented as scientific results. They exist only to prove that the repository organization, configuration system, artifact storage, and deterministic reruns work correctly even before all thesis datasets are integrated. Their use is recorded explicitly in the processed-data manifest so that no reviewer can mistake them for final research data.

## 4. Configuration management and experiment design

Configuration is split into one shared file and two experiment overrides. `config/base.yaml` defines the project-level settings: paths, Python version, seed, preprocessing defaults, output locations, and tracking conventions. The files `experiment_f0.yaml` and `experiment_f1.yaml` extend this base configuration and specify only the experimental differences that matter for comparison.

This design is deliberate. Assignment 4 explicitly requires that experiments are not run by manual editing inside scripts. By placing the feature sets and model parameters in YAML, the repository makes the difference between experiments inspectable and versionable. The baseline experiment `F0` uses `AllAges` and `gp_availability`, while the extended experiment `F1` adds `share_age_65_plus`, `share_female`, `deprivation_index`, and categorical regional controls (`HB`, `HSCP`). The scripts never need to be edited to move from one experiment to the other; only the configuration changes.

The configuration system also supports traceability. Each training run writes out a frozen `config_snapshot.yaml` into the corresponding report folder. This means the exact merged configuration used for the run is stored next to the results, even if the human-readable source file changes later.

## 5. Model pipeline and evaluation logic

The training code in `src/modeling.py` builds a scikit-learn pipeline composed of:

1. numeric imputation,
2. optional numeric scaling,
3. categorical imputation,
4. one-hot encoding for categorical variables,
5. the configured estimator (`ols` or `ridge`).

This structure separates preprocessing from the estimator and guarantees that transformations applied during training are reproduced exactly during evaluation. It also avoids leakage because imputers and encoders are fitted only on the training folds inside the pipeline.

Evaluation follows the experimental logic described in Assignment 3:

- an 80/20 hold-out split,
- 5-fold cross-validation,
- metrics: R², RMSE, and MAE.

The output `metrics.json` stores both hold-out and cross-validation summaries, along with dataset size and the feature lists used. As a result, the reported numbers are not detached from their methodological context.

## 6. Reproducibility and traceability mechanisms

Several concrete engineering decisions were introduced to satisfy reproducibility requirements.

First, dependency versions are pinned in both `requirements.txt` and `environment.yml`, and the Python version is specified as 3.11.9. This avoids ambiguity caused by newer interpreters or incompatible wheels. Second, seed handling is centralized in the configuration, and the runtime explicitly fixes Python and NumPy randomness before training. Third, all paths are resolved relative to the project root so there are no hard-coded absolute machine-specific paths.

The most important traceability mechanism is the deterministic run slug. Each run folder under `reports/runs/` is named from the experiment name, configuration hash, and seed. This gives a stable mapping from configuration to artifact location. The folder contains:

- `metrics.json`,
- `metadata.json`,
- `config_snapshot.yaml`.

In parallel, `models/<run_slug>/model.joblib` stores the serialized estimator, and `reports/experiment_registry.json` stores the global mapping between configuration, run slug, metrics path, model path, and summary metrics. The metadata file also records the processed dataset hash and git commit hash, which is essential for linking thesis results back to the exact code state that generated them.

## 7. Why this design matches the academic requirement

From an academic perspective, the repository now supports software-engineering discipline without pretending to be production infrastructure. The design is intentionally simple, but it enforces the core properties that matter in an engineering thesis:

- explicit structure,
- separation of reusable modules from entry scripts,
- documented environment,
- deterministic execution,
- traceable experiments,
- readable documentation.

This is exactly the transition demanded by Assignment 4: from an informal idea or ad hoc scripts to a controlled implementation environment that can support systematic experimentation in later assignments.

## 8. Limitations and next step toward the final thesis implementation

The current limitation is data availability, not repository organization. The scaffold is ready to ingest the real hospital utilization, workforce, and deprivation tables as soon as they are available locally. At that point the placeholder variables should be removed from preprocessing and replaced with true merged data sources, while keeping the same configuration and artifact conventions. This is an advantage of the chosen design: the core repository architecture does not need to change when the scientific inputs become more complete.

In summary, the delivered repository now satisfies the technical intention of Assignment 4 by making the project reproducible, modular, and auditable, while also documenting clearly where the present scaffold still depends on placeholder data due to missing raw inputs.
